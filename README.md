# CARA — NHS GP Surgery Voice Agent

A production-architecture AI voice receptionist for NHS GP surgeries. Patients call and speak naturally — CARA handles appointment booking, clinical triage, prescription queries, and general admin through a full RAG pipeline with persistent booking state.

Currently configured for a single GP surgery — Greenfield Medical Centre, London — with 4 GPs and a practice nurse, demonstrating how the system would operate at the practice level. The surgery has four GPs — Dr Margaret Collins (diabetes, elderly care), Dr James Whitfield (mental health, skin conditions), Dr Priya Patel (womens health, only female GP, speaks Gujarati and Hindi), and Dr Oliver Bennett (children, respiratory) — plus Practice Nurse Sarah Williams handling routine clinical procedures. The architecture supports multi-tenancy through per-surgery RAG namespaces and separate booking databases.

---

## What CARA Does

- Books, cancels and manages appointments through natural voice conversation
- Triages symptoms and routes emergencies to 999 or 111 immediately
- Handles prescription, registration, sick note, and opening hours queries
- Remembers full conversation context across a call
- Prevents double booking — slots marked unavailable the moment a patient confirms
- Two-factor cancellation verification using reference number and date of birth

---

## Architecture

### RAG Pipeline

Patient queries are classified into one of four intents — appointments, triage, prescription, hours — and routed to dedicated ChromaDB collections. This means triage queries physically cannot return appointment chunks, making dangerous responses architecturally impossible rather than merely unlikely.

Retrieval uses a hybrid approach: dense semantic search using MiniLM embeddings finds chunks with similar meaning, BM25 keyword search finds exact name and term matches, and Reciprocal Rank Fusion combines both ranked lists into a single result without either method dominating.

A cosine relevance threshold gates chunk injection. Below threshold, CARA asks the patient to clarify rather than injecting irrelevant context and generating a confident wrong answer.

### Booking System

Slot filling extracts booking fields from patient speech in any order using a single Claude Haiku call. The patient can give their name first, doctor last, date in the middle — the system handles it.

Date resolution is done entirely in Python. Temporal expressions like "tomorrow", "next Monday", or "end of this week" are resolved to exact calendar dates using datetime arithmetic — never LLM calculation which introduces errors.

All fields are persisted to SQLite immediately as they are collected. A server restart mid-conversation loses nothing. When all required fields are collected, CARA reads back the exact details from the database and asks for an explicit YES or NO — no booking is written until the patient confirms. Idempotency keys prevent duplicate writes even if confirmation fires twice.

Cancellation requires both a reference number and date of birth. Three failed verification attempts locks the session and directs the patient to call reception.

### Clinical Safety

Emergency symptoms route to a dedicated triage collection. A patient saying "I have chest pain and I need an appointment" is classified as triage — the appointment intent is overridden. The system advises 999 immediately. This is enforced at the routing layer, not just the prompt layer.

### Conversation Memory

A rolling summary is generated by Claude Haiku every four turns, capturing what has been resolved and what is still pending. The retrieval query combines this summary with the last six raw turns, giving the retriever both long-term call context and recent specific detail without noise from fully resolved topics.

## Stack

| Component | Technology |
|---|---|
| Backend | FastAPI, Python |
| Vector DB | ChromaDB — 4 intent-routed collections |
| Embeddings | all-MiniLM-L6-v2 (sentence-transformers) |
| Sparse retrieval | BM25Okapi (rank-bm25) |
| Score fusion | Reciprocal Rank Fusion (custom Python) |
| Booking persistence | SQLite with idempotency keys |
| Response generation | Claude Opus 4 |
| Intent classification | Claude Haiku |
| Field extraction | Claude Haiku |
| Conversation summary | Claude Haiku |
| STT / TTS | Web Speech API (production: Whisper + ElevenLabs) |

## Setup

```bash
python -m venv venv
venv\Scripts\activate          # Windows
source venv/bin/activate       # Mac/Linux

pip install -r requirements.txt

# Create .env file and add:
# ANTHROPIC_API_KEY=your_key_here

python rag/ingest.py    # build vector database — run once
python main.py          # start server

# Open Chrome → http://localhost:8000
```

## Project Structure

```
CARA/
├── knowledge_base/
│   ├── doctors.json                 # Clinician profiles, schedules, leave dates
│   ├── appointments.json            # Weekly slot patterns — repeats indefinitely
│   ├── triage_policies.txt          # Clinical triage rules — 999, same day, routine, nurse, self care
│   ├── prescription_policies.txt    # Prescription process and eligibility rules
│   ├── opening_hours.txt            # Surgery hours, out of hours, 111, 999
│   └── faqs.txt                     # Registration, sick notes, referrals, complaints
├── rag/
│   ├── ingest.py                    # Embeds knowledge base into ChromaDB
│   ├── retriever.py                 # Hybrid Dense + BM25 + RRF retrieval
│   ├── intent_classifier.py         # Claude Haiku intent routing
│   ├── relevance_checker.py         # Cosine threshold gate
│   └── conversation_manager.py      # Rolling summary + history management
├── bookings/
│   └── booking_manager.py           # SQLite operations, state machine, cancellation verification
├── static/
│   └── index.html                   # Voice chat UI with live RAG panel
├── main.py                          # FastAPI orchestrator — full pipeline
└── requirements.txt
```
